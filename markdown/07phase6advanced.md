# 07 · Phase 6 — 고급/차별화 (선택 모듈)

## 개요
파이프라인을 "동작하는 것"에서 "잘 설계된 것"으로 끌어올리는 선택 모듈 모음이다. 시간과 목표에 따라 이 중 1~3개를 골라 구현한다. 특히 **Shared Library**와 **Docker 매트릭스**는 "Jenkins를 깊게 다뤘다"는 강한 신호다. 각 모듈은 독립적이라 원하는 것부터 골라 진행하면 된다.

## 선행 조건
- Phase 2~5 완료(트리거·리포트까지 동작하는 파이프라인).

## 모듈 목록
- M1. Docker 동적 에이전트 심화
- M2. 다중 Python 버전 매트릭스 (tox/nox)
- M3. Shared Library로 파이프라인 코드 재사용
- M4. 알림 (Slack/Discord)
- M5. 테스트 통과 시에만 빌드·배포 (deploy on green)
- M6. (선택) 보안 스캔 (bandit)

## 모듈별 상세 동작

### M1. Docker 동적 에이전트 심화
- 이미 Phase 2부터 docker 에이전트를 쓰고 있다면, 여기서 "빌드마다 일회성·격리"를 명확히 보장한다.
- pip 캐시 볼륨, 빌드 의존 OS 패키지 설치를 별도 빌드 이미지(`ci/Dockerfile.agent`)로 분리.
- **동작**: 모든 빌드가 동일·격리 환경에서 실행되어 재현성이 보장된다.

### M2. 다중 Python 버전 매트릭스
- tox(또는 nox)로 3.11·3.12·3.13 환경을 정의(`tox.ini`).
- Jenkins에서 `parallel`로 각 버전을 동시에 실행하거나, Declarative `matrix` 디렉티브 사용.
- **동작**: 한 빌드에서 여러 파이썬 버전 호환성을 동시에 검증한다.
- **완료 신호**: 빌드 그래프에 버전별 병렬 stage가 보이고, 특정 버전에서만 실패하는 케이스를 잡아낼 수 있다.

### M3. Shared Library
- 공통 파이프라인 로직(체크아웃·셋업·리포트 수집)을 별도 저장소/`jenkins/shared-library/vars/`로 추출.
- 예: `vars/pythonTest.groovy`에 표준 테스트 절차를 캡슐화 → `Jenkinsfile`에서 `pythonTest()` 한 줄 호출.
- **동작**: 여러 프로젝트가 동일 파이프라인 표준을 재사용. Jenkinsfile이 짧고 선언적이 된다.
- **포트폴리오 가치**: 가장 높음. "파이프라인을 추상화·재사용했다"는 설계 역량 증명.

### M4. 알림
- Slack Notification 또는 Discord 웹훅으로 빌드 성공/실패를 채널에 전송.
- `post { success { ... } failure { ... } }`에서 호출.
- 메시지에 브랜치·커밋·빌드 URL·테스트 요약 포함.
- **동작**: 빌드 상태를 능동적으로 통지받는다(실무 운영 감각).

### M5. Deploy on Green
- 테스트·게이트가 모두 통과한 경우에만(`when { branch 'main' }` + 성공 조건) 애플리케이션 Docker 이미지를 빌드·태깅·푸시.
- 선택적으로 스테이징 환경에 배포(`docker compose up` 또는 원격 호스트).
- **동작**: 통과 → 산출물(이미지) 생성이라는 CD의 한 조각을 완성. CI에서 CD로 확장.
- **주의**: 레지스트리 자격 증명은 Credentials로만.

### M6. 보안 스캔 (선택)
- `bandit -r app -f xml -o reports/bandit.xml`을 게이트로 추가.
- Warnings NG로 결과 집계.
- **동작**: 정적 보안 취약점(하드코딩 비밀, 위험 함수)을 자동 검출.

## 산출물 (선택한 모듈에 따라)
- `tox.ini` / `nox` 설정 (M2).
- `jenkins/shared-library/vars/*.groovy` + 라이브러리 등록 (M3).
- 알림 설정·자격 증명 (M4).
- `app/Dockerfile`, 레지스트리 푸시 로직 (M5).
- bandit 게이트 (M6).

## 완료 기준 (모듈별)
- [ ] (M2) 여러 파이썬 버전이 병렬로 검증되고, 결과가 버전별로 구분되어 보인다.
- [ ] (M3) `Jenkinsfile`이 공유 라이브러리 호출로 간결해지고, 동일 라이브러리를 재사용 가능.
- [ ] (M4) 빌드 성공/실패 시 채널에 알림이 도착한다.
- [ ] (M5) main 통과 시에만 이미지가 빌드·푸시되고, 실패 시에는 푸시되지 않는다.
- [ ] (M6) 보안 이슈를 주입하면 스캔 게이트가 잡아낸다.

## 함정 / 주의
- **범위 폭발**: 모든 모듈을 다 하려다 미완으로 끝나는 게 최악이다. 1~2개를 끝까지 완성하는 편이 포트폴리오에 낫다.
- **Shared Library 복잡도**: Groovy/CPS 제약(직렬화 가능 변수 등)이 까다롭다. 작은 함수 하나부터 추출한다.
- **매트릭스 비용**: 버전이 늘수록 빌드 시간·자원이 증가. 캐시·병렬 한도(throttle)를 고려.
- **배포 보안**: deploy 단계는 자격 증명·네트워크 노출이 생긴다. 학습 환경에서도 비밀값 노출에 주의.
- **알림 소음**: 모든 빌드마다 알림을 보내면 피로해진다. 실패·main 머지 등 의미 있는 이벤트로 한정.
